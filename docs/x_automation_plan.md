# X投稿の自動化プラン

> マーケティング戦略の一部。全体構成は [CLAUDE.md](../CLAUDE.md) を参照。
> 目的：トレンド連動のX投稿を「生成→承認→投稿」までほぼ全自動で回す。

---

## 全体方針

自動化を2層に分ける。

| 層 | 内容 | 自動化レベル |
|----|------|------------|
| ① 生成 | トレンド収集＋投稿文の作成 | **完全自動** |
| ② 公開 | 実際にXへ投稿 | **1タップ承認**（フェーズ1）→ 安全カテゴリのみ完全自動（フェーズ2） |

理由：実名・代表アカウントのため、時事ネタ連動の投稿は炎上リスクが最も高い。
公開前に人の目を1回通す安全弁を残す。トーンが安定したら段階的に外す。

## アーキテクチャ

```
[スケジュール実行（定期）]
        │
        ▼
① トレンド収集     WebSearchで時事トレンド＋医療制度ニュースを取得
        │
        ▼
② リスク選別       センシティブ話題を自動除外／要注意フラグ付与（下記ルール）
        │
        ▼
③ 投稿文生成       残ったトレンド × 24記事コーパス × 7つの型 で投稿案を作成
        │
        ▼
④ キュー出力       GitHub Issue「X投稿 承認待ち」に投稿案をチェックリストで追記＋古家へ通知
        │
        ▼
⑤ 承認            古家がGitHubアプリ等でIssueを開き、投稿したい案のチェックボックスをタップ（1日10秒）
        │
        ▼
⑥ 投稿            GitHub Actionsがチェック済みの案をX公式APIで投稿し、Issueを更新
```

**承認環境（確定・2026-05-17）：GitHub Issue のチェックボックス方式。**
生成ルーティンが「X投稿 承認待ち」Issueに投稿案をタスクリスト（`- [ ]`）で追加。
古家はGitHubアプリ／Webでチェックボックスをタップ＝承認。GitHub Actionsがチェック済みを投稿。

## リスク選別ルール（時事ネタを扱うため必須）

以下のトレンドには**絶対に絡めない**（自動除外）：

- 事件・事故・犯罪
- 災害・天災
- 訃報・死亡・病気の有名人
- 政治・選挙・政党・国際紛争
- 炎上中の話題、特定個人・団体への批判

**絡めてよい**トレンド：

- 季節・気候（花粉、猛暑、インフル流行期 など）
- 制度・行政（診療報酬改定、特定健診、予防接種事業、IT導入補助金、マイナ保険証 など）
- 一般的な経営・働き方・IT・健康の話題で、医療経営に自然につながるもの

判断に迷うトレンドは「要注意フラグ」を付けて承認者（古家）に必ず回す。

## 投稿の質ルール

- トーン：売り込まない・「先生」呼びかけ・データで説得・短文。[x_posts_stock.md](x_posts_stock.md) の7つの型に準拠。
- 1投稿1メッセージ。【】や絵文字の過剰装飾はしない。
- 医療広告ガイドラインに触れる表現（効果保証・断定）は使わない。事例は「傾向」で書く。
- 週3〜5本。記事リンク付きと単発を交互に。

## 技術構成

- **投稿はX公式API（v2）を使う。** ブラウザ自動化で新規アカウントが定期投稿すると、ボット判定でアカウント凍結リスクが高いため。
- スケジュール実行はクラウドのルーティン（`/schedule`）。ローカルPCに依存しない。
- ルーティンはこのリポジトリで動き、生成結果を `docs/x_queue.md` に追記、承認済みをAPIで投稿する。
- 投稿実行とAPIキーの保管は **GitHub Actions ＋ GitHub Secrets**。

### API料金（重要・2026年2月に無料枠は廃止）

X APIは従量課金（Pay Per Use）。プリペイドのクレジット残高から消費する。

| 操作 | 単価 |
|------|------|
| 通常の投稿 | $0.015 / 件 |
| URLを含む投稿 | $0.20 / 件 |

- 週3〜5本（リンク付き中心）で **月およそ $2〜5（約¥300〜750）**。
- クレジット画面で**支出上限（spending cap）**を設定し、暴走課金を構造的に防ぐこと。

## 投稿頻度・重複防止（確定・2026-05-17）

**頻度：**

| | 頻度 |
|--|--|
| 生成ルーティン | 毎朝1回（投稿案を3本程度） |
| 投稿ワークフロー | 1日3スロット（12:00/16:00/19:00 JST）。承認待ち残数に応じ **0〜3本/日を自動調整** |
| 実ペース | 残数1〜2→1本/日、3〜4→2本/日、5以上→3本/日。承認量・ストックに自動追従 |

**重複防止（5段構え）：**

1. 投稿案ごとに一意のID（例 `gen-20260518-1`）
2. 投稿済みIDを台帳 `x-bot/posted.json` に記録（gitコミット＝確実に残る）
3. 投稿ワークフローは「☑承認済み かつ 台帳に未記録」のIDのみ投稿
4. 投稿成功の直後に台帳へ記録＆コミット
5. 二重の保険：X API自体が同一本文の連続投稿を拒否／Actionsの `concurrency` で二重起動防止

生成ルーティンも `posted.json` を読んでから生成し、内容のかぶりを避ける。
