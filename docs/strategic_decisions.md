## 1. 事業仮説の検証

### 「古家はレアな存在か？」

**結論：レアだが、レア＝勝てるではない。**

4層フィルター（業界接触 → 不の認知 → 解決策設計 → 実行）を通過した希少人材である点は事実。ただし市場の存在・顧客の課題認識・実行の質と速度という別の論点がある。

### 「ミルカルテに金を払う人はいるか？」

**結論：払う人はいる。ただし簡単には見つからない。**

- 課題は実在する（院長は患者離脱を感覚的に認識している）
- 優先度が低い（緊急ではない領域、売り込みでは動かない）
- 支払い意思は条件付き（成果が見えた瞬間に財布が開く）

**設計思想は正しい**（3層構造：可視化 → 郵便代行 → LINE運用代行）。郵便・LINE運用代行こそがロックインの本体で、可視化単体での課金は弱い。

## 2. 開発 vs 営業の並列方針（確定）

**「リアル先行 vs 開発先行」ではなく、並列で走らせる。**

- 従来型の指摘：課題解像度の重要性（AI時代でも変わらない）
- 古家の視点：AIで検証サイクルが速くなった（開発コスト低下）
- 現場学習は画面越しでは代替不可能（院長の反応、事務スタッフの操作の詰まり）

**採用方針：** リアル1院アプローチを開始 ＋ AI MVP開発を並行。「完成してからリアルへ」ではなく「リアルで聞いた課題を即実装」のループ。

## 4. 身元開示・ブランディング方針（確定）

### 顔出し・実名で発信する

- 医療領域では匿名は信用されない
- 医療データを扱うサービスで匿名開発者は無理
- 実名・顔・現場経験が揃った個人は市場に数えるほどしかいない

### OASIS名は出さない

- 利益相反の疑義を構造的に回避
- 副業規定との衝突を回避
- 守秘情報流出リスクがゼロ
- 独立時にブランドが残る（ポータビリティ）

**失うもの：** 初見の信用の足場、肩書による即時リーチ加速
**カバー手段：** 経験の抽象化、バーバラ企画代表の肩書、技術スタックの明示

### リスク対策

- OASIS業務と完全分離
- 副業規定の事前確認
- 患者視点の言葉選び（「離脱を取り戻す」→「通院継続を支援する」）
